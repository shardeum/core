# AccountPatcher Debugging Guide

## Overview

The AccountPatcher system includes extensive debugging capabilities through stats objects, counters, and logging mechanisms. This guide provides a structured reference for understanding and utilizing these debugging tools to diagnose issues with account synchronization and repair.

## Stats Objects

### 1. `HashTrieUpdateStats`

Generated by `upateShardTrie()` to track trie update performance.

```typescript
type HashTrieUpdateStats = {
  updatedNodesPerLevel: number[]      // Nodes updated at each trie level
  hashedChildrenPerLevel: number[]    // Children hashed at each level
  totalHashes: number                  // Total hash operations performed
  totalAccountsHashed: number          // Individual accounts processed
  totalNodesHashed: number             // Trie nodes with computed hashes
  totalLeafs: number                   // Leaf nodes in the trie
}
```

**Interpretation**:
- High `totalAccountsHashed`: Many account changes this cycle
- Uneven `updatedNodesPerLevel`: Localized changes in address space
- `totalLeafs` growth: Network size increasing

### 2. `AccountHashStats`

Used in hash comparison operations.

```typescript
type AccountHashStats = {
  matched: number           // Nodes matching expected hash
  visited: number          // Total nodes examined
  empty: number            // Null/empty nodes encountered
  nullResults: number      // Null responses from queries
  numRequests: number      // Network requests made
  responses: number        // Successful responses
  exceptions: number       // Errors during processing
  radixToReq: number      // Radixes queued for requests
  actualRadixRequests: number  // Actual network requests sent
}
```

**Key Ratios**:
- `matched/visited`: Hash match rate (should be high when in sync)
- `responses/numRequests`: Network reliability
- `actualRadixRequests/radixToReq`: Request deduplication efficiency

### 3. `BadAccountStats`

Core stats from `findBadAccounts()` operation.

```typescript
type BadAccountStats = {
  testedSyncRadix: number     // Sync radixes examined
  skippedSyncRadix: number    // Radixes skipped (syncing)
  badSyncRadix: number        // Radixes needing repair
  needsVotes: number          // Radixes lacking consensus
  ok_noTrieAcc: number        // OK despite no trie account
  ok_trieHashBad: number      // OK but trie hash mismatch
  fix_butHashMatch: number    // Needs fix but hashes match
  fixLastSeen: number         // Fixed last seen cycle
  subHashesTested: number     // Child hashes examined
  trailColdLevel: number      // Depth where trail went cold
  checkedLevel: number        // Maximum level checked
  leafsChecked: number        // Leaf nodes examined
  leafResponses: number       // Leaf data responses
  getAccountHashStats: {}     // Nested hash stats
}
```

**Critical Indicators**:
- `badSyncRadix > 0`: Out of sync with network
- `needsVotes > 0`: Insufficient consensus data
- `trailColdLevel < treeMaxDepth`: Incomplete traversal
- `skippedSyncRadix` high: Many ranges being synced

### 4. `FilterStats`

Tracks repair data filtering in `testAndPatchAccounts()`.

```typescript
type FilterStats = {
  accepted: number          // Updates accepted for application
  tooOld: number           // Updates rejected as outdated
  sameTS: number           // Same timestamp (usually skipped)
  sameTSFix: number        // Same timestamp but different hash
  tsFix2: number           // Secondary timestamp fixes
  tsFix3: number           // Tertiary timestamp fixes
}
```

**Warning Signs**:
- `tooOld > 0`: Node has newer data than majority
- `tooOld > accepted`: Majority behind this node
- All zeros: No viable repairs found

### 5. `IsInsyncResult`

Result of sync check operation.

```typescript
type IsInsyncResult = {
  radixes: string[]        // Radixes checked
  insync: boolean         // Overall sync status
  stats: {
    good: number          // Radixes in sync
    bad: number           // Radixes out of sync  
    total: number         // Total radixes checked
  }
}
```

**Health Indicators**:
- `insync = true`: Node synchronized
- `bad/total` ratio: Sync quality (should be 0)
- `total = 0`: No consensus data available

## Counter Events

The AccountPatcher uses `nestedCountersInstance.countEvent()` extensively. Key counter patterns:

### Sync and Consensus Counters

```javascript
// Sync level operations
'accountPatcher', 'isInsync-radix-length-mismatch'     // Invalid radix length
'accountPatcher', `not enough votes ${radix}`          // Insufficient consensus
'accountPatcher', 'findBadAccounts hasNonConsensusRange' // Non-consensus range detected
'accountPatcher', 'findBadAccounts hasNonStorageRange'   // Non-storage range detected

// Broadcast operations  
'accountPatcher', `broadcastSyncHashes c:${cycle}`, count  // Hashes broadcast
'accountPatcher', `sync_trie_hashes-send`, messageCount    // Messages sent
```

### Repair Counters

```javascript
// Account repair flow
'accountPatcher', `badAccounts c:${cycle}`, count          // Bad accounts found
'accountPatcher', `accountHashesChecked c:${cycle}`, count // Accounts examined
'accountPatcher', `checkAndSetAccountData updateTooOld c:${cycle}` // Newer local data
'accountPatcher', `removedAccounts c:${cycle}`, count      // Accounts removed
'accountPatcher', `removedAccountsFailed c:${cycle}`, count // Failed removals

// Repair outcomes
'repair', `filter.start c:${cycle}`, count                 // Filtering started
'repair', `filter.hasAccount c:${cycle}`                   // Account exists
'repair', `filter.rejected.tooOld c:${cycle}`             // Rejected as old
'repair', `filter.accepted.existing c:${cycle}`           // Accepted existing
'repair', `filter.accepted.new c:${cycle}`                // Accepted new
```

### Network Communication Counters

```javascript
// Binary endpoint usage
'internal', InternalRouteEnum.binary_sync_trie_hashes      
'internal', InternalRouteEnum.binary_get_trie_hashes
'internal', InternalRouteEnum.binary_get_trie_accountHashes
'internal', InternalRouteEnum.binary_repair_oos_accounts

// Response tracking
'accountPatcher', `get_trie_hashes c:${cycle}`, responseCount
'accountPatcher', `get_trie_accountHashes c:${cycle}`, accountCount
```

## Debug Logging

### Fatal Logs

Critical issues logged via `statemanager_fatal()`:

```javascript
// Sync state debugging
'debug shardTrie', `temp shardTrie votes c:${cycle}: ${votes}`
'debug findBadAccounts', `toFix: ${radixes}`

// Repair issues  
'checkAndSetAccountData updateTooOld', `${accountId} updateTS:${ts1} storageTS:${ts2}`
'checkAndSetAccountData extra bad keys', `c:${cycle} keys: ${keys}`
'checkAndSetAccountData repairDataResponse', `c:${cycle} repairDataResponse is null`
```

### Main Logger

Standard debug logging via `mainLogger.debug()`:

```javascript
// Sync status
`isInSync: cycle: ${cycle}, isInsyncResult: ${result}`

// Bad account detection
`badAccounts cycle: ${cycle}, ourBadAccounts: ${count}`
`findBadAccounts ${cycle}: level: ${level}, toFix: ${count}`

// Repair filtering
`Starting repair filter: cycle ${cycle}, ${count} accounts`
`Account ${id} exists in cache/storage: storageTS:${ts1}, wrappedTS:${ts2}`
`REJECTED (too old): Account ${id}, repair timestamp ${ts1} is ${diff}ms older`
`ACCEPTED (existing): Account ${id}, repair timestamp ${ts1} >= storage ${ts2}`
```

## Diagnostic Flows

### 1. Checking Sync Status

```javascript
// Look for these patterns:
isInsyncResult.insync === false                    // Out of sync
isInsyncResult.stats.bad > 0                      // Specific radixes bad
stats.needsVotes > minVotes                       // Insufficient consensus
stats.testedSyncRadix vs stats.badSyncRadix       // Sync quality ratio
```

### 2. Identifying Repair Issues

```javascript
// Check repair effectiveness:
filterStats.tooOld > 0                            // Have newer data
filterStats.accepted === 0                        // No valid repairs
badAccounts.length > 0 but appliedFixes === 0    // Repairs failing
tooOldAccountsMap.size > 0                       // Reverse healing needed
```

### 3. Network Communication Health

```javascript
// Monitor request/response patterns:
stats.responses < stats.numRequests              // Network issues
stats.exceptions > 0                             // Processing errors
actualRadixRequests >> radixToReq               // Inefficient batching
```

### 4. Trie Structure Health

```javascript
// Evaluate trie integrity:
updateStats.totalLeafs                          // Network size proxy
incompleteNodes.length > 0                      // Coverage gaps
stats.trailColdLevel < treeMaxDepth            // Incomplete traversal
```

## Common Debugging Scenarios

### Scenario 1: Node Stuck Out of Sync

**Symptoms**:
- `isInsyncResult.insync = false` persists
- `failedLastTrieSync = true` repeatedly
- `syncFailHistory` growing

**Investigation**:
1. Check `stats.needsVotes` - insufficient consensus?
2. Examine `filterStats.tooOld` - newer data locally?
3. Review `stats.skippedSyncRadix` - too many syncing ranges?
4. Verify network connectivity via response rates

### Scenario 2: High Repair Traffic

**Symptoms**:
- Large `badAccounts.length`
- High `repairRequestsMadeThisCycle.numRequests`
- Network congestion

**Investigation**:
1. Check `stats.badSyncRadix` distribution
2. Look for `hasNonConsensusRange` events
3. Examine `updateStats.totalAccountsHashed` for churn
4. Review cycle timing for delays

### Scenario 3: Repairs Not Applied

**Symptoms**:
- `badAccounts.length > 0`
- `filterStats.accepted = 0`
- Same accounts repeatedly flagged

**Investigation**:
1. Check `filterStats.tooOld` for timestamp issues
2. Examine `tooOldAccountsMap` entries
3. Look for `checkAndSetAccountData` failures
4. Verify receipt validation logic

### Scenario 4: Incomplete Coverage

**Symptoms**:
- `incompleteNodes.length > 0`
- `stats.trailColdLevel` stops early
- Missing account data

**Investigation**:
1. Check coverage ranges via `getNonConsensusRanges()`
2. Examine `radixIsStored` map
3. Look for syncing neighbors
4. Verify shard calculations

## Performance Metrics

### Key Performance Indicators

```javascript
// Efficiency metrics
const syncEfficiency = stats.matched / stats.visited
const repairSuccessRate = filterStats.accepted / (filterStats.accepted + filterStats.tooOld)
const consensusStrength = (stats.testedSyncRadix - stats.needsVotes) / stats.testedSyncRadix
const networkReliability = stats.responses / stats.numRequests

// Load metrics
const accountChurn = updateStats.totalAccountsHashed / updateStats.totalLeafs
const repairLoad = badAccounts.length / updateStats.totalLeafs
const trieDepthEfficiency = stats.checkedLevel / treeMaxDepth
```

### Warning Thresholds

- `syncEfficiency < 0.95`: Potential sync issues
- `repairSuccessRate < 0.5`: Majority behind this node
- `consensusStrength < 0.8`: Weak network consensus
- `networkReliability < 0.9`: Communication problems
- `accountChurn > 0.1`: High state change rate
- `repairLoad > 0.05`: Significant repair burden

## Advanced Debugging

### Trace Analysis

For deep debugging, combine multiple signals:

```javascript
// Full repair trace
const repairTrace = {
  cycle,
  syncStatus: isInsyncResult,
  badAccountStats: stats,
  filterResults: filterStats,
  repairRequests: repairRequestsMadeThisCycle,
  failHistory: syncFailHistory,
  tooOldAccounts: tooOldAccountsMap.size,
  appliedFixes: updatedAccounts.length
}
```

### Correlation Patterns

Look for these correlations:
- `needsVotes` high + `skippedSyncRadix` high = Network transition
- `tooOld` high + `badSyncRadix` low = Leading the network
- `exceptions` high + `responses` low = Network partition
- `trailColdLevel` varies + `incompleteNodes` high = Coverage issues

### Historical Analysis

Track trends over time:
- `syncFailHistory`: Patterns of failure
- `failRepairsCounter`: Cumulative repair attempts
- Cycle-over-cycle `badAccounts.length`
- `updateStats.totalAccountsHashed` trends

## Logging Best Practices

### Development Mode
- Enable all debug flags: `logFlags.debug = true`
- Track individual account repairs
- Log full vote maps and consensus details
- Capture complete repair traces

### Production Mode
- Focus on counter events
- Log only repair summaries
- Track failure patterns
- Monitor performance metrics

### Troubleshooting Mode
- Enable `statemanager_fatal` logs
- Increase verbosity for specific cycles
- Capture network message details
- Log repair filtering decisions

## Summary

The AccountPatcher's debugging infrastructure provides comprehensive visibility into the synchronization and repair process. By understanding the relationships between different stats objects, counter events, and log patterns, operators can effectively diagnose and resolve synchronization issues. The key is to look for patterns across multiple indicators rather than relying on single metrics, as the distributed nature of the system means that temporary inconsistencies are normal and only persistent patterns indicate real issues.
